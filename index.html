Containers

Containers are lightweight, standalone, executable packages of software that contain (heh) everything needed in order to run your app.
  • Code
  • Runtime
  • System Libraries
  • etc...
Kubernetes is a container orchestrator to automate the process of deploying and managing multi-container applications at scale.
Kubernetes is not a replacement for Docker. Kubernetes works with Docker and any container system that conforms to the Open Container Initiative (OCI).
So why Kubernetes?
  • Multi-container deployments
  • Scale containerized applications
  • Rolling updates without downtime
  • Networking, service discovery, and storage
  • Use any environment (Cloud, Bare Metal)
  • Manages app health and loading balancing
  • Manages resource allocation

How it works

Etcd
This is simply a key value store that persists the desired state.
Controller Manager
This is a control loop that watches the actual state and works to get it to match the desired state.
Scheduler
This component uses various configurations and live metrics to determine the best place to place a pod.
Scheduler Example
API Server
This is how most of the communication happens between clients.
DNS
This is how service names are able to resolve to pods.

Abstractions

Cluster
This is the highest-level abstraction and refers to all of the machines running K8. Clusters can be Federated to support a multi cloud architecture.
  • Master(s)
  • Node(s)
Namespaces
These are separate environments or ecosystems within the same cluster.
  • default
  • kube-system
Pods
A grouping of one or more containers that make up an application. These are groupings that you would want to be able to horizontally scale.
Containers
The lowest-level abstraction. This is usually an immutable image containing the code that does one thing really well, but doesn't need to scale independently.
Controllers
Further abstractions to represent the type of things you can run on K8.
  • Deployments
  • Jobs
  • Cron Jobs
  • Stateful Sets
  • Daemon Sets
Networking
K8 handles service discovery, load balancing, and networking for us.
  • Services
  • DNS
  • Ingress
  • Network Policies
Storage
In general, you want to keep your containers stateless, but when you do need persistance there's a thing for it.
  • Persistent Volumes
  • Persistent Volume Claims

Examples YAY!

Deployments
A way to manage replica sets or a given number of pod replicas.
Deployments 
          
						apiVersion: apps/v1
						kind: Deployment
						metadata:
							name: impress-deployment
							labels:
								app: impress
						spec:
							replicas: 3
							selector:
								matchLabels:
									app: impress
							template:
								metadata:
									labels:
										app: impress
								spec:
									containers:
									- name: devin-test
										image: devinburnette/devin-test:latest
										ports:
										- containerPort: 80
Services
A service makes a pod or group of pods accessible by exposing ports (internally or externally).
Services 
          
            apiVersion: v1
            kind: Service
            metadata:
              name: impress-service
            spec:
              type: LoadBalancer
              selector:
                app: impress
              ports:
              - protocol: TCP
                port: 8000
                targetPort: 80
